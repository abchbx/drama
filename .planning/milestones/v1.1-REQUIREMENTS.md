# Requirements: v1.1 — Routing, Memory & Integration

**Defined:** 2026-03-18
**Archived:** 2026-03-20

## v1.1 Requirements

### Phase 5: Message Routing Hub

- [x] **ROUTE-01**: Socket.IO server implements broadcast routing (Director → all actors)
- [x] **ROUTE-02**: Socket.IO server implements peer-to-peer routing (Actor → Actor)
- [x] **ROUTE-03**: Socket.IO server implements multicast routing (Director → subset of actors)
- [x] **ROUTE-04**: Heartbeat signals: agents confirm alive status periodically
- [x] **ROUTE-05**: Every message wait has explicit timeout with defined fallback behavior
- [x] **ROUTE-06**: Director has unilateral fallback when actors do not respond within timeout

### Phase 6: Memory Management Engine

- [x] **MEM-01**: Automated token counting on every blackboard write with 60% threshold alert
- [x] **MEM-02**: Semantic layer folding: when >8K tokens, summary replaces individual entries
- [x] **MEM-03**: Procedural layer folding: when >4K tokens, character voice constraints are preserved
- [x] **MEM-04**: Core layer is NEVER evicted — only explicitly updated
- [x] **MEM-05**: Director explicitly promotes scenario content to core layer when plot-critical

### Phase 7: Integration + Chaos Testing

- [x] **TEST-01**: End-to-end drama session test: Director + 2+ Actors complete a scene from start to finish
- [x] **TEST-02**: Chaos test: boundary violation attempts are rejected with appropriate error
- [x] **TEST-03**: Chaos test: silent agent (no response) triggers timeout fallback
- [x] **TEST-04**: Chaos test: concurrent blackboard writes are handled without data loss
- [x] **TEST-05**: Chaos test: message reorder does not cause narrative inconsistency
- [x] **TEST-06**: Token budget overflow test: session continues correctly after semantic layer fold

### Cross-Cutting

- [x] **PROTO-01**: JSON message protocol with speaker identification, cognitive_state field, and scene phase
- [x] **PROTO-02**: Message types formally defined (Zod schema) before implementation
- [x] **PROTO-03**: LLM Provider abstraction: OpenAI and Anthropic adapters behind unified interface
- [x] **PROTO-04**: All configuration via .env file (API keys, ports, token budgets, timeouts)
- [x] **PROTO-05**: Structured JSON logging with agent attribution on every log line

---

## Requirement Outcomes

All v1.1 requirements have been met. The system now includes:
- **Socket.IO real-time messaging** with broadcast, peer-to-peer, and multicast capabilities
- **Four-layer memory management** with automated folding, token budget enforcement, and core preservation
- **Comprehensive integration testing** with 104 tests covering E2E scenarios and chaos conditions
- **LLM provider abstraction** with OpenAI and Anthropic adapters behind a unified interface
- **Formalized communication protocol** using Zod schemas, .env configuration, and structured logging

## Validated from v1.0

- [x] **BLKB-01**: Shared blackboard service exposes REST API for read/write operations
- [x] **BLKB-02**: Four-layer memory model (core/scenario/semantic/procedural) with hard token budgets per layer
- [x] **BLKB-03**: Optimistic locking with version vectors on all blackboard writes
- [x] **BLKB-04**: Audit log: every write attributed to agent ID, timestamp, and message ID
- [x] **BLKB-05**: JSON file snapshot persistence for blackboard state
- [x] **BOUND-01**: Write-layer gate rejects entries that violate capability closure (actors cannot write to core layer)
- [x] **BOUND-02**: Namespace isolation: Actor agents can only read their character card and current scene
- [x] **BOUND-03**: Per-agent input scope restriction enforced at the blackboard API layer
- [x] **BOUND-04**: Boundary enforcement is hard (programmatic), not soft (prompt-based)
- [x] **ACTR-01**: Actor agents generate dialogue constrained by their character card
- [x] **ACTR-02**: Actor agents have scoped blackboard read access (character card + current scene only)
- [x] **ACTR-03**: Hallucination flag system marks unverified details in blackboard entries
- [x] **ACTR-04**: Character voice consistency: character voice stored as explicit style constraints in procedural layer
- [x] **ACTR-05**: Director acts as character consistency auditor and fact-checks scenes against core layer
- [x] **DIR-01**: Director agent plans plot backbone and writes to blackboard core/scenario layers
- [x] **DIR-02**: Director arbitrates key decisions when Actor outputs conflict
- [x] **DIR-03**: Director MUST NOT monopolize creative authority — explicit role contract enforced
- [x] **DIR-04**: Intentional holes: Director mandates "Actor discretion" scenes where actors introduce creative content
- [x] **DIR-05**: Director fact-checks all scenes against established core layer facts
- [x] **DIR-06**: Director enforces inter-phase validation: validates Actor output at every phase transition
